# 多 Agent 协作标注方案 V3

> 思路：Python 控制多轮调用，每个 Agent 负责单一任务，模拟多 Agent 独立上下文。
> 模型：qwen3:32b（能力优先于上下文大小）
> 旧项目 baseline：87.99% 准确率

---

## 1. 核心思路

不是让模型自己决定调不调子 Agent（旧项目证明它不会调），而是 **Python 强制执行多轮调用**，每轮只问模型一个窄问题。

```
Python 控制流（每轮标注）：
  第1轮 → 场景师       → 分析场景（读±15-20行）
  第2轮 → 角色师       → 判断说话人（读±10-15行）
  第3轮 → 矛盾师       → 后文搜索身份揭示（读后续50行）
           └── 如果没找到 → 压缩师压缩这50行 → 矛盾师继续往后看
  第4轮 → 记忆师       → 汇总短期记忆 + 长期记忆
  第5轮 → 标注师       → 综合以上所有信息，最终判断
```

每个 Agent：
- **独立上下文**——只看到自己需要的那部分，不浪费 token
- **单一任务**——不需要思考流程，只需要回答问题
- **Python 强制执行**——不存在"模型不调用"的问题

## 2. Agent 分工

### 2.1 场景师（Scene Analyzer）

| 属性 | 说明 |
|------|------|
| 输入 | 对话行前后各 15-20 行原文 |
| 任务 | 用一句话概括当前是什么场景 |
| 输出示例 | `村民在卖皮草，罗伦斯在询价` |

```
prompt: "以下是小说片段。请用一句话概括当前场景（谁在哪里、在做什么）：
         [上下文]
         场景概括："
```

### 2.2 角色师（Character Analyzer）

| 属性 | 说明 |
|------|------|
| 输入 | 对话行前后各 10-15 行原文 + 当前对话内容 |
| 任务 | 列出当前最可能参与对话的角色（1-3个） |
| 输出示例 | `可能角色：村民（说话者）、罗伦斯（被称呼者）` |

```
prompt: "以下是小说片段。请分析这段对话的说话人是谁（只输出名字）：
         上下文：[...]
         待标对话：「...」
         说话人："
```

### 2.3 矛盾师（Lookahead Analyzer）

| 属性 | 说明 |
|------|------|
| 输入 | 当前行往后 50 行原文 + 压缩师的历史压缩信息 |
| 任务 | 检查后文是否有身份揭示 |
| 输出 | `后续第X行揭示：说话者是村民` 或 `后续无身份揭示` |

如果 50 行内无身份揭示，Python 调用压缩师压缩这 50 行，然后矛盾师（重置上下文）带着压缩信息继续看下一个 50 行。如此循环，直到找到身份揭示或到达搜索上限（如 500 行）。

```
矛盾师第1轮：读第50-100行 → 没找到
  → 压缩师压缩第50-100行 → "罗伦斯继续赶路"
矛盾师第2轮：读第100-150行 + 压缩信息 → 没找到
  → 压缩师压缩第100-150行 → "罗伦斯遇到巡逻的骑士"
矛盾师第3轮：读第150-200行 + 压缩信息 → 找到了！
```

### 2.4 压缩师（Compressor）

| 属性 | 说明 |
|------|------|
| 输入 | 一段原文（50行） |
| 任务 | 用一句话浓缩这段内容，保留角色名和关键事件 |
| 输出示例 | `罗伦斯遇到骑士，两人交谈后骑士离开` |

```
prompt: "请用一句话概括以下小说段落中发生的事情（保留角色名）：
         [50行原文]
         概括："
```

### 2.5 记忆师（Memory Agent）

| 属性 | 说明 |
|------|------|
| 输入 | 短期记忆（最近N段对话的总结）+ 长期记忆（全卷总结）|
| 任务 | 汇总与当前对话相关的历史信息 |
| 输出示例 | `短期：刚才村民在卖皮草。长期：罗伦斯是旅行商人，赫萝是贤狼` |

### 2.6 记忆更新师（Memory Updater）

| 属性 | 说明 |
|------|------|
| 调用时机 | 每标注完 1 段对话 |
| 任务 | 判断是否需要更新短期记忆 |
| 规则 | 如果当前对话引入了新信息（新角色、新场景、剧情转折），更新短期记忆 |
| 更新方式 | 旧短期记忆 + 当前对话场景摘要 = 新短期记忆 |

```
prompt: "当前短期记忆：{旧短期记忆}
         当前对话场景：{场景师结果}
         请判断是否需要更新短期记忆。如果需要，用一句话概括新的短期记忆："
```

### 2.7 长期记忆更新师（Long-term Memory Updater）

| 属性 | 说明 |
|------|------|
| 调用时机 | 每标注完 20 段对话（可配置） |
| 任务 | 用当前短期记忆更新长期记忆 |
| 更新方式 | 旧长期记忆 + 当前短期记忆 = 新长期记忆 |

```
prompt: "旧长期记忆：{旧长期记忆}
         最近进展：{当前短期记忆}
         请用一句话概括更新后的长期记忆："
```

### 2.8 角色库维护师（Registry Keeper）

| 属性 | 说明 |
|------|------|
| 输入 | 当前标注的说话人 + 行号 |
| 任务 | 追加到 characters.json |
| 实现 | **纯 Python 字典操作**，不需要调模型 |

### 2.9 角色归一师（Character Normalizer）

| 属性 | 说明 |
|------|------|
| 输入 | 当前标注的说话人名字 + 角色注册表中的所有角色名 |
| 任务 | 判断当前名字是否与已有角色重复（别名/简称/不同叫法）|
| 输出示例 | `贤狼赫萝 → 已有角色“赫萝”，合并` 或 `叶勒 → 新角色，新建条目` |
| 调用时机 | 每次写入 labeled.txt 之后 |

```
prompt: "当前标注了说话人：{说话人}
         已有角色：{角色列表}
         请判断：这个说话人是否与已有角色是同一人？
         如果是同一人，输出已有角色名。
         如果是新角色，输出 NEW。"
```

纯 Python 无法判断「赫萝」和「贤狼赫萝」是同一人，但大模型一眼就能看出来。这个 Agent 就是为了解决角色名不一致的问题。

### 2.10 标注师（Final Labeler）

| 属性 | 说明 |
|------|------|
| 输入 | 场景师 + 角色师 + 矛盾师 + 记忆师 的结果汇总 |
| 任务 | 综合所有信息，输出最终说话人 |
| 输出示例 | `村民` |

```
prompt: "你是一名最终标注师。以下是多个分析维度汇总的信息：
         场景：{场景师结果}
         角色分析：{角色师结果}
         后文调查：{矛盾师结果}
         历史信息：{记忆师结果}

         请综合以上信息，确定这段对话「{对话内容}」的说话人是谁。
         只输出说话人名字，不要多余内容。
         说话人："
```

## 3. 数据流

```
                    Python 主循环
                         │
                    ┌────▼────┐
                    │ 读取下一段对话 │
                    └────┬────┘
                         │
              ┌──────────┼──────────┐
              ▼          ▼          ▼
         场景师     角色师     矛盾师 ──→ 压缩师
         (读±20行)  (读±15行)  (读后续50行)  (搜不到时压缩)
              │          │          │          │
              │          │          ▼          │
              │          │    (循环直到找到    │
              │          │     或达到上限)     │
              │          │                    │
              └──────────┼────────────────────┘
                         ▼
                     记忆师
              (短期记忆 + 长期记忆)
                         │
                         ▼
                     标注师
              (综合所有信息判最终)
                         │
                         ▼
               ┌─────────┼─────────┐
               ▼         ▼         ▼
         角色库维护   记忆更新师   写 labeled.txt
         (纯Python)  (更新短期)   (追加一行)
                         │
                         ▼
                  长期记忆更新师
                  (每20条更新一次)
                         │
                         ▼
                    更新进度 → 继续下一轮
```

## 4. 记忆系统

### 4.1 记忆结构

| 记忆类型 | 存储位置 | 内容 | 更新时机 |
|---------|---------|------|---------|
| **短期记忆** | `.omo/memory/short_term.json` | 最近 N 段对话的浓缩总结（1-2句话） | 每 1-2 条标注后更新 |
| **长期记忆** | `.omo/memory/long_term.json` | 全卷至今的浓缩总结（2-3句话） | 每 20 条标注后更新 |
| **角色注册表** | `.omo/evidence/characters.json` | 所有出现过的角色名 + 首次行号 + 出现次数 | 发现新角色时（纯Python） |

### 4.2 记忆演进过程

```
第1-5条：
  短期记忆 = "开头：村民在村庄集市与旅行商人罗伦斯交易皮草"
  长期记忆 = 同短期记忆（初始一致）

第6-20条：
  短期记忆 = "罗伦斯离开村庄后遇到骑士，两人结伴同行"
  长期记忆 = "第1章：罗伦斯在村庄收购皮草后离开，途中遇到骑士"

第21-40条：
  短期记忆 = "罗伦斯和骑士抵达帕兹欧，赫萝初次登场"
  长期记忆 = "第1-2章：罗伦斯从村民手中收购皮草，途中遇到骑士并结伴，
              到达帕兹欧小镇后遇到自称贤狼的少女赫萝"

第41-100条：
  短期记忆 = "赫萝自称贤狼，与罗伦斯达成交易一同旅行"
  长期记忆 = "第1-3章：罗伦斯收购皮草→遇到骑士→到达帕兹欧→
              与赫萝相遇并结伴→赫萝自称贤狼、展现超常智慧"
```

### 4.3 更新规则

更新时机不由固定行数决定，而是由**对话间距**决定。

**判断依据**：当前对话行号与上一条对话行号的差值。

```
对话1  (L23) ─┐
对话2  (L24) ─┤  间距1-3行 → 同一场景 → 不更新短期记忆
对话3  (L25) ─┤
对话4  (L26) ─┘
               │
对话5  (L41) ──┘  间距15行 → 新场景 → 触发短期记忆更新
```

**短期记忆更新**（场景切换时触发）：
```
if 当前行号 - 上条行号 > 阈值(如5行):
    # 场景切换了，更新短期记忆
    记忆更新师收到：旧短期记忆 + 刚结束的场景摘要
    → 压缩融合成新短期记忆
else:
    # 同一场景，不更新
```

这样短期记忆自然地在场景边界处更新，而不是机械地每隔 N 条更新一次。

**长期记忆更新**：仍然以短期记忆的版本号为依据。每更新 5-10 次短期记忆后，触发一次长期记忆更新。

## 5. 与旧项目的对比

| 对比项 | opencode-novel-loop | 本方案 V3 |
|--------|-------------------|-----------|
| 模型 | qwen3:32b | qwen3:32b（同一模型，公平对比） |
| 控制方式 | Python 单轮调用 | Python 多轮调用（模拟多 Agent） |
| 上下文 | 只读一次上下文 | 每个 Agent 读不同范围的上下文 |
| 后文搜索 | 有（locate_identity + resolve_identity） | 有（矛盾师） |
| 角色库 | 有（主 Agent 自行维护） | 有（纯 Python 维护，可靠） |
| 短期记忆 | 无 | 有（最近 5 条标注） |
| 准确率 baseline | 87.99% | 目标：≥ 88% |

## 6. 需要进一步讨论的问题

1. **Agent 数量**：5 个 Agent 是否太多？可以精简吗？
2. **模型选择**：所有 Agent 都用 qwen3:32b，还是标注师用 qwen3:32b 其他用 4b？
3. **矛盾师的搜索范围**：后续 50 行够吗？还是需要更大的范围？
4. **角色库维护**：纯 Python 追加，还是也需要模型参与？
5. **记忆师的角色库维度**：除了角色名和次数，还需要记录什么（角色关系、口癖等）？
